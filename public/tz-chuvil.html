🗺 О проекте

Многостраничный сайт + CMS + маркетплейс для дизайнера интерьеров. Три роли пользователей:

  • admin — дизайнер/владелец. Полный доступ ко всему.
  • customer — заказчик. Видит только свой проект.
  • contractor — подрядчик. Видит свои задания и профиль.

⚙️ Стек

Nuxt 3 TypeScript Drizzle ORM PostgreSQL Redis (ioredis) Minio (S3) Pinia Nuxt UI Tailwind CSS 4 Zod jose (JWT) bcryptjs sharp pdfmake Resend (email)

Пример: вход

// Запрос
fetch('/api/auth/login', {
  method: 'POST',
  body: JSON.stringify({ login: 'admin@mail.ru', password: 'secret123' })
})

// Ответ 200
{ "userId": "uuid...", "role": "admin", "name": "Дарья" }

// Ошибка 429 (rate-limit — 5 попыток за 15 мин)
{ "statusMessage": "Too many login attempts. Try again in 14 minutes." }

Защита API-роутов

import { requireSession, requireRole } from '~/server/utils/auth'

// Любой авторизованный пользователь
const session = await requireSession(event)
// → { userId, role, ... } или автоматически 401

// Только admin
const session = await requireRole(event, 'admin')

// Admin или contractor
const session = await requireRole(event, ['admin', 'contractor'])

Защита страниц (Nuxt middleware)

// В компоненте страницы:

definePageMeta({ middleware: 'contractor' })  // только contractor
definePageMeta({ middleware: 'require-auth' }) // любой авторизованный

🖊️ Модуль 4 — CMS и генерация HTML

Module 4 · Статус: готов

CMS-редактор страниц + fs.writeFile генерация

Редактор JSON-контента в админке. Сохраняет в JSONB, генерирует статический HTML через шаблонизатор.

API: server/api/cms/
Шаблонизатор: server/utils/html-generator.ts
UI: app/pages/admin/cms/

Как это работает

  1. Admin открывает /admin/cms → выбирает проект и тип страницы
  2. В редакторе видит JSON-шаблон, заполняет данными
  3. Нажимает Сохранить → PUT /api/cms/content/[projectId]/[slug] → JSONB в PostgreSQL
  4. Нажимает Сгенерировать HTML → POST /api/cms/generate/[projectId]/[slug]
  5. Сервер читает JSONB, прогоняет через шаблон, пишет файл → public/generated/[slug-проекта]/[тип].html
  6. Ссылка на файл возвращается в ответе и открывается по клику

6.3 Два способа загрузки

Способ 1 — Через сервер (multipart): просто, но файл проходит через наш Node.js сервер. Хорошо для малых файлов (до 5 MB) — сервер сжимает через sharp.

POST /api/upload
Content-Type: multipart/form-data

file=<файл>  context=material_photo  projectId=uuid

Способ 2 — Presigned URL (прямо в Minio): для больших файлов (Го + пдф). Файл не проходит через наш сервер.

1. GET /api/upload/presign?filename=photo.jpg&mimeType=image/jpeg&context=work_photo
   ← { uploadUrl: "https://minio:9000/daria-files/...", storageKey: "...", expiresIn: 900 }

2. PUT {uploadUrl}   ← клиент льёт напрямую в Minio

3. POST /api/upload/confirm  { storageKey, originalName, mimeType, sizeBytes, context }
   ← UploadResult (запись в БД создана)

5.11 Защита от инъекций и атак на данные

Файл: server/middleware/05.injection-guard.ts — запускается на каждый API-запрос до обработчика.

Дополнения в: server/utils/validate.ts, server/utils/sanitize.ts

Почему это нужно, если Drizzle ORM уже защищает?
Drizzle параметризует SQL-запросы — это главная защита. Но атаки бывают разные: через URL-параметры, через заголовки, через структуру JSON. Middleware перехватывает их до того как любой код успел обработать запрос.

1. SQL-инъекции

Что это: злоумышленник подставляет SQL-команды в поля ввода, чтобы прочитать или удалить базу данных.

// Плохой пример без защиты:
GET /api/contractors?search='; DROP TABLE users; --

// После внедрения в raw SQL это стало бы:
SELECT * FROM contractors WHERE name ILIKE '%'; DROP TABLE users; --%'
// ☠️ Таблица users удалена

// Как защищаемся:
// 1. Drizzle ВСЕГДА параметризует: ilike(users.name, `%${search}%`)
//    → PostgreSQL получит: ILIKE $1, параметр: '%search_text%' — команда не выполнится
// 2. Middleware дополнительно блокирует грубые паттерны в query-строке

Паттерны которые блокируются: UNION SELECT, DROP TABLE, DELETE FROM, INSERT INTO, EXEC(, xp_cmdshell, pg_sleep(, WAITFOR DELAY, ; DROP (stacked queries), -- в конце строки.

2. Null-byte инъекция (нулевой байт)

Что это: символ \x00 (нулевой байт) вставляется в строку. PostgreSQL и некоторые C-библиотеки обрезают строку по нулевому байту — это может обойти LIKE-фильтры или проверки прав.

// Атака: подрядчик ищет проект чужого клиента
GET /api/projects?search=ремонт%00' OR '1'='1

// PostgreSQL TEXT обрезает по \x00 → строка становится просто "ремонт"
// Потом ' OR '1'='1 может выполниться как SQL если где-то есть raw запрос

// Защита: middleware удаляет все \x00 из URL, query и тела запроса
// sanitize.ts:  stripControlChars() убирает \x00 из всех строк

3. CRLF-инъекция (инъекция в заголовки)

Что это: символы \r\n (перенос строки в HTTP) вставляются в данные. Позволяет добавлять поддельные HTTP-заголовки в ответ или поддельные записи в логи.

// Атака: злой IP-адрес в X-Forwarded-For
X-Forwarded-For: 1.2.3.4\r\nSet-Cookie: admin=true

// Без защиты этот IP попал бы в Redis-ключ:  rl:write:1.2.3.4\r\nSet-Cookie: admin=true
// Redis может интерпретировать \r\n как разделитель команд → command injection

// Защита: sanitizeIp() в validate.ts оставляет ТОЛЬКО цифры/точки/двоеточия
// "1.2.3.4\r\nDEL *"  →  "1.2.3.4"  (Redis-ключ безопасен)

4. Prototype Pollution (загрязнение прототипа)

Что это: атака через JSON-тело. Если в объект через __proto__ добавить свойство — оно появится у всех объектов в Node.js процессе. Один запрос может сделать всех пользователей администраторами.

// Атака:
POST /api/projects
{ "__proto__": { "isAdmin": true }, "title": "мой проект" }

// Если сервер делает Object.assign({}, body) или spread {...body}
// то у ВСЕХ объектов появится isAdmin: true до перезапуска сервера
// if (user.isAdmin) → true для всех пользователей ☠️

// Защита: middleware проверяет каждый ключ JSON рекурсивно
// Ключи __proto__ / constructor / prototype → сразу 400 Bad Request

5. Log Injection (инъекция в логи)

Что это: пользователь передаёт строку с переносами строк \n. Если сервер логирует её напрямую — в журнале появятся поддельные записи. Это маскирует реальные атаки и вводит в заблуждение при аудите.

// Атака:
email = "user@evil.com\n[2026-01-01 03:00] INFO: admin logged in from 127.0.0.1"

// Без защиты в логах появится:
// [2026-02-28] Login attempt: user@evil.com
// [2026-01-01 03:00] INFO: admin logged in from 127.0.0.1  ← поддельная строка

// Защита: sanitizeLogString() заменяет \n \r \t → пробел перед Console.log
// console.info('[auth] login:', sanitizeLogString(email))

6. IP-спуфинг (обход rate-limit)

Что это: заголовок X-Forwarded-For можно подделать. Без проверки злоумышленник может обойти ограничение попыток входа, просто меняя значение этого заголовка.

// Атака: брутфорс с обходом rate-limit
POST /api/auth/login  →  X-Forwarded-For: 1.1.1.1   (попытка 1-5, заблокирован)
POST /api/auth/login  →  X-Forwarded-For: 2.2.2.2   (попытка 6-10, новый "IP")
POST /api/auth/login  →  X-Forwarded-For: 3.3.3.3   (и так далее...)

// Защита: в production nginx настраивается как единственный trusted proxy
// При настройке nginx → $real_ip используется вместо X-Forwarded-For клиента
// sanitizeIp() убирает цепочки IP и невалидные значения → 'unknown' один ключ на всех

7. Аномально длинные параметры (попытки переполнения / ReDoS)

Что это: очень длинная строка в URL-параметре может вызвать зависание regex (ReDoS) или переполнить буфер. Middleware обрезает параметры больше 2000 символов — этого хватает для любого реального поиска.
