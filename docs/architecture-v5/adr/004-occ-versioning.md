# ADR-004: Optimistic Concurrency Control через поле `version`

## Контекст

Десктопный и мобильный клиенты могут работать в режиме плохой связи (offline/low-latency). При восстановлении соединения оба пытаются записать изменения в одну и ту же запись. Без механизма обнаружения конфликтов одно из изменений молча перезаписывает другое — пользователь теряет работу.

Пессимистичные блокировки (SELECT FOR UPDATE) не подходят: они держат транзакцию открытой, несовместимы с REST-семантикой и неприемлемы при высоком concurrency.

## Решение

**Optimistic Concurrency Control (OCC)** через целочисленное поле `version`:

- Каждая изменяемая таблица имеет колонку `version integer not null default 1`.
- При чтении клиент получает текущий `version`.
- При UPDATE/PATCH клиент отправляет `version` обратно.
- Server выполняет:
  ```sql
  UPDATE <table> SET ..., version = version + 1
  WHERE id = $id AND version = $client_version AND deleted_at IS NULL
  ```
- Если `rowsAffected = 0` — запись не найдена или version устарел → сервер возвращает `409 Conflict`.
- Клиент обрабатывает 409: показывает diff, предлагает merge или перезагрузку.

Все новые таблицы в `messenger_*` (Doc-23) следуют тому же паттерну. Подробности: [23 Project-Centric Messenger §2](../23-project-centric-messenger.md).

## Статус

Принято. Зафиксировано как обязательный инвариант в [05 Architectural Patterns §5.4](../05-architectural-patterns.md) и [CLAUDE.md](../../../../CLAUDE.md) (правило №8).

## Последствия

- **Клиент должен обрабатывать 409** — это не ошибка сервера, а сигнал о конфликте. Frontend обновляется на чтение `error.code === 'CONFLICT'`.
- **Все мутируемые строки нуждаются в `version`** — DDL-требование для всех новых таблиц.
- **Незначительный write overhead** — дополнительная проверка в WHERE; пренебрежимо мало для OLTP-нагрузки.
- **Нет автоматического merge** — конфликты разрешает пользователь или клиент-сторонняя логика.
- **Soft-delete совместим**: `deleted_at IS NULL` в WHERE гарантирует, что update удалённой записи вернёт 0 rowsAffected → 409 (или 404, на усмотрение handler'а).
