# 7. Микросервис Мессенджера (Standalone)

Модуль чатов и WebRTC вынесен за пределы основного монолита Nuxt в директорию `messenger/`.

## Почему Standalone?
Основной сервер (Nuxt H3) занимается рендерингом страниц (SSR) и тяжелыми REST вычислениями. Постоянно висящие WebSocket соединения (и WebRTC сигналы) при росте онлайна блокируют Event Loop. Выделение мессенджера в отдельный процесс с собственным PM2-воркером решает эту проблему.

## Структура `messenger/`

- **`core/` (Ядро сервера)**:
  - WebSocket сервер (на базе `ws` или `socket.io`).
  - Логика распределения комнат (Rooms).
  - **Redis Subscriber**: Подключение к шине `ioredis` для прослушивания внутренних событий от основного API.
- **`web/` (Фронтенд чата)**:
  - Может содержать независимую сборку чат-клиента (например, для встройки через IFrame), либо специфичные страницы (login/chat).
- **`ecosystem.standalone.config.cjs`**: 
  - Конфигурация запуска для PM2, отделенная от основного `daria-nuxt`.

## Интеграция с Nuxt (Backend)
1. **Авторизация**: Реализуется через паттерн *Ticket-Based Auth*. Nuxt проверяет cookie, выдает одноразовый токен в Redis, Messenger считывает его при открытии WS-соединения. Мы не дублируем логику сессий.
2. **Шина данных (Pub/Sub)**:
   - Событие на бекенде: Пользователь загружает файл -> Nuxt сохраняет в Drizzle -> Nuxt делает `redis.publish('chat', JSON.stringify({ event: 'new_msg' }))`.
   - Реакция мессенджера: Messenger слушает `redis.subscribe('chat')` -> Видит событие -> Рассылает его нужным клиентам через открытые WS.
