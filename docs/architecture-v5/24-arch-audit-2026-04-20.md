# 24. Project-Centric Messenger: Architecture Audit (2026-04-20)

Audit scope: `docs/architecture-v5/23-project-centric-messenger.md` vs. `messenger/core/src/projects/**`, `messenger/web/app/entities/projects/**`, `messenger/web/app/features/**`, `messenger/web/app/widgets/**`.

**Date**: 2026-04-20  
**Status**: 7 dimensions audited; critical gaps found in OCC layer and routing.

---

## Summary

- ✗ **OCC missing across 5 resource tables** — `version` column not present in connectors, skills, plugins, MCP, external APIs despite spec §2 mandate.
- ✗ **Route collision** — both `[projectSlug].vue` and `[projectId].vue` pages exist, competing for the same intent.
- ✗ **API response shape drift** — handler returns `{ mcp: ... }`, frontend types expect `{ server: ... }`.
- ✗ **MCP ping endpoint missing** — backend handler lacks `/projects/:id/mcp/:entryId/ping` route.
- ✗ **Hard delete on skills/plugins** — spec requires soft-delete + OCC; implementation uses hard delete.
- ✗ **No MVP secrets TODO** — plaintext secret storage not flagged; spec §7 red line unmet.
- ~ **Skills/plugins upsert semantics** — backend correctly implements onConflictDoUpdate; frontend support unclear.

## Naming collisions

1. **Page route duplication** (`messenger/web/app/pages/projects/`):
   - `[projectSlug].vue` (uses `route.params.projectSlug`, calls `api.getProject(projectSlug)`)
   - `[projectId].vue` (uses `route.params.projectId`, calls `api.getProject(projectId)`, has `definePageMeta({ path: '/projects/by-id/:projectId' })`)
   - **Both render MessengerProjectWorkspace** with identical logic; spec clearly intends slug-based routing.
   - **Fix needed**: Remove `[projectId].vue` or clarify intent if ID-based fallback is required.

2. **Response field naming inconsistency**:
   - Handler (`projects-handler.ts:511`): returns `{ mcp: entry }`
   - API types (`app/core/api/projects.ts:48`): types expect `{ server: MessengerMcpServer }`
   - External API endpoint (`projects.ts:66`): declares return as `{ apis: ... }` but handler returns `{ externalApi: ... }`

## Missed Zod validations

**Status: Minimal gap.** All POST/PATCH endpoints have Zod schemas. However:

1. **Bootstrap JSON parsing** (`projects-handler.ts:788`): lenient regex-based JSON extraction.
   ```ts
   const jsonMatch = rawReply.match(/\{[\s\S]*\}/)
   ```
   Spec §7 says "Zod-схема обязательна" — schema is present (`proposalSchema` line 116), but fallback to raw text on parse failure is loose.

2. **Connector config** (`connectorBody.ts:64`): `z.record(z.unknown()).optional()` — no validation of secret refs or required fields per connector type (e.g., `claude-cli` should enforce `sessionSlugPool` structure).

## Error-response drift

**Status: Minor inconsistencies.** Error responses follow `{ error: <CODE>, details?: string }` pattern.

Drifts observed:
- `NOT_FOUND` used generically for skills/plugins, but `PROJECT_NOT_FOUND` used for projects.
  - Line 347, 362, 423 (skills/plugins): `{ error: 'NOT_FOUND' }`
  - Line 274, 291, 318, etc. (projects): `{ error: 'PROJECT_NOT_FOUND' }`
- **No functional issue** (error codes are distinct), but naming inconsistency.

## OCC & soft-delete audit

**Critical finding**: OCC not implemented for non-project resources.

| Table | OCC (`version`) | Soft-delete (`deleted_at`) | Status |
|---|---|---|---|
| `messengerProjects` | ✓ (`integer version, default 1`) | ✓ | **SPEC-COMPLIANT** |
| `messengerProjectConnectors` | ✗ | ✓ | **OCC MISSING** |
| `messengerProjectSkills` | ✗ | ✗ (hard delete) | **OCC + SOFT-DELETE MISSING** |
| `messengerProjectPlugins` | ✗ | ✗ (hard delete) | **OCC + SOFT-DELETE MISSING** |
| `messengerProjectMcp` | ✗ | ✓ | **OCC MISSING** |
| `messengerProjectExternalApis` | ✗ | ✓ | **OCC MISSING** |

**Evidence**:
- Schema: `server/db/schema/messenger-project-*.ts` — no `version` column in any resource table.
- Update handlers: connectors/MCP/external-APIs update without version check; skills/plugins lack version entirely.
- Hard-delete: `project-resource-store.ts:192-200` (skills), `:250-258` (plugins) call `db.delete(...)` instead of `db.update(...).set({ deletedAt: ... })`.

**Spec citations**:
- §2: "OCC через `version`, soft-delete через `deleted_at`"
- §6 Phase acceptance: "миграция применима и ревертится" implies schema must support it
- §7 Red line: "(later: OCC version everywhere)" — suggests all mutable rows need it

## Secrets handling

**Status: Unmarked plaintext storage.** Spec §2.2 and §7 Red line state:

> Secrets are NEVER stored in config jsonb; only refs. … for MVP we store plaintext in config with a clear TODO and guard via row-level access.

**Findings**:
- Connector config stored as-is in `config` jsonb, no validation or warning.
- No `messenger_secrets` table (planned Phase 2) exists.
- No TODO comment in schema, handler, or store warning about MVP plaintext mode.
- Spec example shows `{ secretRef: 'secret_xxx' }`, but implementation allows any `config: Record<string, unknown>`.

**Compliance**: Missing the "clear TODO" per spec §7 red line. Plaintext secrets in production would be a compliance issue.

3. **Resolve page routing collision** — remove `[projectId].vue` or clearly document as fallback-only route.
4. **Fix API response field names** — align handler return names (`mcp` vs `server`, `externalApi` vs `api`) across all endpoints.
5. **Implement MCP ping endpoint** — add `POST /projects/:id/mcp/:entryId/ping` handler (health check logic TBD).
6. **Add secrets MVP TODO** — mark plaintext secret storage with explicit TODO comment in schema and handler for Phase 2 migration.
7. **Validate connector config per type** — enforce required fields (e.g., `claude-cli` expects `sessionSlugPool: string[]`).
8. **Complete MessengerProject type** — add `version`, `icon`, `color` to frontend composable interface.
