# 16. Playbook расширения: новые микросервисы и приложения

Этот документ отвечает на один вопрос: **как добавить в монорепозиторий новый рантайм, не сломав архитектуру v5**.

Он дополняет:
- [01. Инфраструктура](./01-infrastructure.md) — таблица рантаймов и портов.
- [02. Структура монорепозитория](./02-monorepo-structure.md) — пять правил изоляции.
- [05. Архитектурные паттерны](./05-architectural-patterns.md#59-horizontal-extensibility) — три паттерна интеграции.

## 16.1. Когда выносить в отдельный рантайм

Критерии выноса (достаточно одного, чтобы обосновать):

1. **Event-loop блокировка**: нагрузка держит постоянные соединения (WebSocket, WebRTC, SSE-broadcasts) или тяжело работает с CPU (шифрование, ML-инференс).
2. **Security-домен**: сервис имеет право не знать схему основной БД (E2EE-реле, внешние интеграции с третьими API).
3. **Независимый жизненный цикл**: отдельный продуктовый сайт со своими релизами, своей UI-системой, своим доменом (пример: `cityfarm/web`).
4. **Язык/платформа**: сервис лучше писать на Go/Python/Rust, а не на Node (например, ML-воркер).
5. **Масштабирование**: нужно уметь горизонтально масштабировать именно эту функцию, не масштабируя Nuxt-app.

Если ни один из критериев не выполняется — функция должна жить внутри `server/modules/<domain>/` главного Nuxt-app.

## 16.2. Три паттерна интеграции

| Паттерн              | Когда применять                          | Как общается с main app            |
|----------------------|------------------------------------------|------------------------------------|
| HTTP-consumer        | Синхронные запросы, не-realtime          | REST в `server/api/**`             |
| Pub/Sub-listener     | Фоновые задачи, реакция на события       | Redis `SUBSCRIBE <channel>`        |
| Ticket-based realtime| WebSocket/WebRTC для конечных клиентов   | Одноразовый токен из Redis         |

Сервис может комбинировать паттерны (например, messenger/core использует Pub/Sub-listener + Ticket-based realtime), но **никогда не подключается напрямую к Postgres основного app**.

## 16.3. Чек-лист добавления нового микросервиса

Пример: добавляем `services/billing-service/`.

### Шаг 1. Решение о выносе
- [ ] Выбран и задокументирован критерий из 16.1.
- [ ] Выбран паттерн/паттерны интеграции из 16.2.
- [ ] Согласован с командой (PR с обновлением 02 и 16).

### Шаг 2. Каркас каталога
```text
services/billing-service/
├── package.json              # Своё имя, свои зависимости
├── tsconfig.json             # Расширяет root tsconfig или изолирует
├── ecosystem.config.cjs      # Отдельный PM2-блок (port 3030+)
├── README.md                 # Чем сервис занимается и как запускать
└── src/
    ├── index.ts              # Bootstrap: config + http/ws server + shutdown hooks
    ├── config.ts             # Env + валидация через Zod
    ├── auth/                 # Если нужен ticket-based auth
    ├── store/                # Если нужно своё состояние (pg-store/redis-store)
    └── <domain>/             # Bounded-context директории
```

- [ ] Добавлен в `pnpm-workspace.yaml`.
- [ ] Добавлен в `docker-compose.yml` (если деплоится в контейнере).
- [ ] Добавлен в корневой `ecosystem.config.cjs` и при необходимости в `ecosystem.refactor.config.cjs`.

### Шаг 3. Контракты
- [ ] Все DTO сервиса объявлены в `shared/types/<domain>/**`.
- [ ] Все enum/статусы объявлены в `shared/constants/<domain>/**`.
- [ ] Валидация входа — через Zod-схемы из `shared/types/**`.
- [ ] Для Pub/Sub: имена каналов и событий — в `shared/constants/system/websocket-events.ts` (единственный source of truth).

### Шаг 4. Безопасность
- [ ] Сервис **не импортирует** `server/db/**`.
- [ ] Сервис **не читает** сессионные cookies главного app.
- [ ] Для realtime: использует `verifyTicket(ticket)` из Redis, тикет генерируется в `server/modules/auth/session.service.ts`.
- [ ] Логи не содержат PII/секретов; для E2EE-сервисов — не логируется `content`.
- [ ] `/health` эндпоинт возвращает 200 при готовности.

### Шаг 5. Документация
- [ ] Обновлена таблица рантаймов в `01-infrastructure.md`.
- [ ] Дерево обновлено в `02-monorepo-structure.md`.
- [ ] Добавлена секция в `12-messenger-services-refactor-map.md` (или новый файл, если сервис не realtime).
- [ ] Создан файл `.github/instructions/<service-name>.instructions.md` с правилами Copilot для этого слайса.
- [ ] Обновлён `.github/copilot-instructions.md` (таблица scope → файл).

### Шаг 6. Валидация перед merge
- [ ] `pnpm --filter <service> build` — ok.
- [ ] `pnpm --filter <service> typecheck` — ok.
- [ ] Сервис поднимается локально и отвечает `/health`.
- [ ] Main app продолжает работать без нового сервиса (graceful degradation).

## 16.4. Чек-лист добавления нового публичного приложения (site-style)

Пример: добавляем `studio-blog/web/` по образцу `cityfarm/web/`.

### Шаг 1. Каркас
```text
studio-blog/
├── web/
│   ├── package.json          # Свой Nuxt
│   ├── nuxt.config.ts        # Свои modules/runtimeConfig
│   ├── app/                  # Свой FSD-срез (core/shared/entities/...)
│   ├── server/               # Свой Nitro (если нужен свой API)
│   └── public/
└── ecosystem.config.cjs      # Отдельный PM2-блок
```

### Шаг 2. Изоляция
- [ ] Приложение **не импортирует** из главного `app/**` или `server/**`.
- [ ] Если нужны общие типы/константы — они сначала выносятся в `shared/**` и только потом используются.
- [ ] Собственная UI-система допустима (cityfarm использует свою), но внутри приложения она должна быть консистентной.

### Шаг 3. Деплой
- [ ] Отдельный nginx-server_name / отдельный поддомен.
- [ ] Отдельный PM2-блок со своим портом (см. 01).
- [ ] Отдельный `deploy-<app>.sh` в `scripts/` при нетривиальной логике сборки.

### Шаг 4. Документация
- [ ] Обновлено `02-monorepo-structure.md`.
- [ ] Обновлена таблица в `01-infrastructure.md`.
- [ ] Создан `.github/instructions/<app>.instructions.md` с ограничениями UI-системы.

## 16.5. Антипаттерны (что делать нельзя)

1. ❌ **Добавлять новый сервис внутрь `server/api/**`** как ещё одну папку endpoint-ов «чтобы быстрее». Если сервис подпадает под 16.1 — он выносится в `services/`.
2. ❌ **Подключать новый сервис напрямую к Postgres** main app. Только через HTTP/Pub-Sub/ticket.
3. ❌ **Импортировать код из `app/` или `messenger/web/app/` в новое приложение.** Только через `shared/**`.
4. ❌ **Дублировать типы в `services/<new>/src/types.ts`**, если они уже есть в `shared/types/**`.
5. ❌ **Переиспользовать порт** существующего рантайма или выбирать порт вне диапазона 30xx (основной), 34xx (публичные сайты).
6. ❌ **Запускать новый сервис без `/health` эндпоинта** — делает невозможным rolling restart через PM2.
7. ❌ **Добавлять нового рантайма без записи в `01-infrastructure.md`** — ломает операционную прозрачность.

## 16.6. Быстрая шпаргалка

| Задача                                     | Куда класть                                       |
|--------------------------------------------|---------------------------------------------------|
| CRUD по новой сущности CRM                 | `server/modules/<entity>/` + `server/api/<entity>/` |
| Фоновый воркер с Redis-очередью            | `services/<name>-worker/` (Pub/Sub-listener)      |
| Публичный сайт-лендинг                     | `<name>/web/` в корне                             |
| Новый канал связи (SMS, email provider)    | `services/<name>-service/` (HTTP-consumer)        |
| Новое WebSocket-приложение                 | `services/<name>-service/` (Ticket-based realtime)|
| Новая UI-страница в CRM                    | `app/pages/` + `app/widgets/**` (не новый рантайм)|
| Новая страница в мессенджере               | `messenger/web/app/pages/**` (не новый рантайм)   |

Если сомневаешься — функция скорее всего должна жить в `server/modules/` или `app/features/`, а не в новом рантайме. Вынос в отдельный процесс — это архитектурное решение, а не оптимизация кода.
