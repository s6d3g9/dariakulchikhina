# 1. Глобальная инфраструктура и Развертывание

Система разворачивается через изолированные контейнеры. Прямые загрузки файлов на диск (`public/uploads`) запрещены. Прямые подключения к БД от сервисов запрещены (только через пул).

## Основные компоненты (Docker Compose)
- **PostgreSQL 16**: Основная реляционная база данных.
- **PgBouncer**: Пул соединений для PostgreSQL. Все микросервисы (Nuxt, AI, Messenger) ходят только через него (режим transaction).
  > **Внимание (Миграции):** Drizzle-миграции (`drizzle-kit push/migrate`) выполняются строго в обход PgBouncer (по прямому URL), так как миграции требуют DDL-транзакций, которые пул может прервать.
- **Redis 7**: Выполняет 3 роли:
  1. Pub/Sub (шина событий между Nuxt и Messenger).
  2. Кэш (тикеты авторизации, тяжелые списки проектов, конфигурации UI/тем).
  3. Очередь задач (BullMQ для фоновых задач и AI).
- **MinIO (S3)**: Объектное хранилище для всех файлов, чертежей и фото.
- **Coturn (TURN/STUN)**: Сервер для пробития NAT при WebRTC видеозвонках.

## Мониторинг и Логирование

> **Статус:** целевая модель. Частично реализовано через локальные JSON-логи PM2; централизованный sink ещё не развернут.

- **Loki / ELK** *(планируется)*: Все сервисы пишут логи в JSON через общий logger.
- **Trace IDs** *(частично)*: Сквозная трассировка запросов (`X-Request-ID`) между Nuxt → Redis → AI-worker → БД.
- **Sentry** *(планируется)*: Отлов ошибок на фронтенде и в Nitro.
- **Health-эндпоинты:** каждый отдельный рантайм (Nuxt-app, messenger/core, communications-service, cityfarm/web, будущие сервисы) обязан экспортировать `/health` с кодом 200 при готовности.

## Развертывание и CI/CD

- **Миграции БД**: Drizzle migrations выполняются в обход PgBouncer.
- **Кэширование сборки**: Использование `pnpm fetch` / Turborepo для монорепозитория.
- **Zero-Downtime**: Rolling updates через PM2 Cluster mode или Docker Swarm.
